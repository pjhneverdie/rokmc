# 프론트 컨트롤러
mvc를 적용한 서블릿 방식은 반복 코드가 너무 많이 발생. 그래서 공통 처리를 위해 프론트 컨트롤러라는 대표 서블릿을 앞에 두는 구조.
프론트 컨트롤러에서 각 요청 url이랑 해당 요청을 처리할 컨트롤러를 매치해서 요청을 처리하는 것임. 

이렇게 했을 때 프론트 컨트롤러만 서블릿이면 되고 나머지 컨트롤러는 그냥 자바 클래스로 만들면 되니까 객체지향 관점에서, 이전처럼 모든 컨트롤러가 다 서블릿을 구현할 필요가 없으니 더 이상 종속적이지 않게 되는 장점도 있음.

기본 mvc 골자에서 아래처럼 간다고 보면 됨.
```java
@WebServlet("/*")
public class FrontControllerServlet extends HttpServlet {

    protected void service(HttpServletRequest req, HttpServletResponse resp) {
        // 스프링 아니니까 controllerMap 노가다로 작성
        Controller controller = controllerMap.get(req.getRequestURI());
        controller.process(req);
    }
}

```
```java
public interface Controller {
    // 이전에는 모든 컨트롤러가 HttpServlet를 구현해야 했음.
    // 지금도 맵핑 편의를 위해 Controller를 만들어서 구현해야 하긴 하지만 HttpServlet 보다 훨 가벼움
    String process(HttpServletRequest req);
}
```
```java
public class UserController implements Controller {

    @Override
    public String process(HttpServletRequest request) {
        List<User> users = userService.findAll();

        request.setAttribute("users", users);

        return "/WEB-INF/views/userList.jsp";
    }
}
```

# 왜 안 쓰냐
Controller 한 개당 process 메서드 하나니까 UserListController, UserFormController, UserSaveController, UserDeleteController 이런 식으로 클래스가 너무 많이 필요.
지금 스프링은 UserController에서 싹 처리해버리잖씀.
